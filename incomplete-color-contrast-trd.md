# Technical Requirements Document: A11y-Oracle Visual Contrast Engine

**Automated Remediation for `axe-core` Incomplete Contrast Violations**

## 1. Executive Summary

The A11y-Oracle Visual Contrast Engine is a specialized testing utility designed to resolve "Incomplete / Needs Review" color contrast warnings generated by DOM-based accessibility scanners like `axe-core`. When text is placed over CSS gradients, background images, or absolutely positioned layers, static scanners cannot mathematically verify the background color.

This engine intercepts those specific `incomplete` results, applies strict CSS "Halo" heuristics, and utilizes headless browser screenshotting coupled with pixel-level relative luminance analysis. By mathematically verifying the worst-case and best-case contrast scenarios within the text's bounding box, the engine safely filters out false positives and escalates true WCAG 1.4.3/1.4.6 violations, significantly reducing alert fatigue in CI/CD pipelines.

## 2. System Architecture (Nx Workspace)

This tool will be housed within an Nx monorepo utilizing two primary libraries dedicated to visual analysis:

- **`libs/axe-bridge`**: A middleware function that consumes the raw JSON output from `axe-core`. It isolates the `incomplete` array, filters for `id === 'color-contrast'`, and orchestrates the visual evaluation pipeline.
- **`libs/visual-engine`**: A Node.js utility utilizing an image buffer processor (e.g., `sharp`). It handles CSS parsing for text shadows and calculates the Relative Luminance of pixel arrays extracted from headless browser screenshots.

## 3. The "Incomplete" Resolution Pipeline

When a test suite runs, the pipeline executes the following automated protocol:

1. **Payload Interception:** `libs/axe-bridge` intercepts the `axe-core` results, isolating nodes flagged as `incomplete` due to `color-contrast`.
2. **The Halo Check:** The engine first evaluates the node for valid CSS text strokes or shadows (detailed in Section 4). If a valid halo passes contrast, the issue is filtered out.
3. **DOM Manipulation (Text Hiding):** If no halo exists, the engine captures the computed foreground text `color`. It then injects CSS to visually hide the text stroke without altering the layout: `color: transparent !important; text-shadow: none !important; caret-color: transparent !important;`
4. **Visual Capture:** The test runner (Playwright/Cypress) captures a bounding-box screenshot of the element, which now displays _only_ the background gradient or image.
5. **Pixel Analysis:** `libs/visual-engine` scans the image buffer to find the pixels with the highest ($L_{max}$) and lowest ($L_{min}$) relative luminance.
6. **Restoration:** The transparent CSS is removed, restoring the page state.

## 4. The Halo Heuristic (Pre-Processing)

WCAG explicitly allows a text halo or stroke to act as the contrast boundary. Before performing expensive image processing, the engine applies strict regex and CSS parsing to validate if a CSS shadow qualifies as a compliant halo.

- **Valid Stroke:** If `-webkit-text-stroke-width` is $\ge$ 1px, and `-webkit-text-stroke-color` passes the 4.5:1 ratio against the background visual pixels, the node is marked as **Pass**.
- **Valid Shadow:** To count a `text-shadow` as a valid halo, it must uniformly surround the text. The engine parses the CSS string. If the shadow has a blur radius of 0 and stacks in multiple directions (e.g., `1px 1px 0 #000, -1px -1px 0 #000...`) to cover the glyph, and that color passes, the node is marked as **Pass**.
- **Invalid Shadow:** Any shadow with a blur radius $> 0$ (a drop shadow or glow) is immediately rejected as a halo, and the pipeline proceeds to visual background analysis.

## 5. Visual Pixel Analysis & Contrast Math

To ensure strict WCAG 2.1 AA compliance, the `visual-engine` adheres exactly to the W3C formulas for Relative Luminance and Contrast Ratio.

The engine calculates the contrast ratio ($CR$) between the foreground text color ($L_{text}$) and the extreme pixels of the background bounding box:

$$CR = \frac{L_{lighter} + 0.05}{L_{darker} + 0.05}$$

**The Algorithm Logic:**

1. Calculate $L_{text}$ for the foreground text color.
2. Scan the background screenshot buffer to find $L_{max}$ (lightest pixel) and $L_{min}$ (darkest pixel).
3. Run the $CR$ formula twice:

- $CR_{1}$ using $L_{text}$ and $L_{max}$
- $CR_{2}$ using $L_{text}$ and $L_{min}$

## 6. The Safe Assessment Matrix

To absolutely prevent False Positives (failing a component that is actually readable) or False Negatives (passing a component that is unreadable), the engine categorizes the result based on a strict matrix.

If the engine cannot be 100% certain, it retains the `Incomplete` status.

| Condition              | Mathematical Result                                                                                        | Final Tool Output | Action Taken                                              |
| ---------------------- | ---------------------------------------------------------------------------------------------------------- | ----------------- | --------------------------------------------------------- |
| **Passes Worst-Case**  | Passes against the _most conflicting_ background pixel.                                                    | **✅ Pass**       | Filtered out. Not reported.                               |
| **Fails Best-Case**    | Fails against the _most favorable_ background pixel.                                                       | **❌ Violation**  | Sent to reporting dashboard. (Unreadable everywhere).     |
| **Split Decision**     | Passes favorable pixel, but fails worst-case pixel (e.g., left-aligned text on a black-to-white gradient). | **⚠️ Incomplete** | Retained as incomplete. Sent to QA/Dev for manual review. |
| **Dynamic / Temporal** | Element sits over `<video>`, `<canvas>`, or uses `< 1` alpha opacity/blend modes.                          | **⚠️ Incomplete** | Retained as incomplete. Sent to QA/Dev for manual review. |

## 7. API Integration Example

The developer experience allows teams to seamlessly wrap their existing `axe-core` implementation.

```javascript
import { A11yOracle } from '@a11y-oracle/axe-bridge';

test('Page meets contrast standards including gradients', async ({ page }) => {
  // 1. Run standard Axe-Core
  const axeResults = await new AxeBuilder({ page }).analyze();

  // 2. Pass results through the Visual Contrast Engine
  const cleanResults = await A11yOracle.resolveIncompleteContrast(
    page,
    axeResults,
    {
      threshold: 4.5, // WCAG AA Standard (3.0 for large text)
    }
  );

  // 3. Assert zero violations
  expect(cleanResults.violations.length).toBe(0);

  // 4. (Optional) Log remaining true "incompletes" for manual QA
  console.log(cleanResults.incomplete);
});
```
